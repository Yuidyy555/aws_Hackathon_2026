# ユーザーストーリー（MVP）

## ストーリー分解方針
- **採用方式**: Hybrid（Journey + Feature）
- **粒度**: 標準（8〜12本）
- **受け入れ条件レベル**: 標準（正常系 + 主要エラー系）

## Epic 1: 入力から変換までの中核体験

### US-01 感情と文脈の入力
**As a** エンジニアユーザー  
**I want** 文脈テキストと感情を短く入力したい  
**So that** 余計な負荷なく変換を依頼できる

**Acceptance Criteria**
1. 文脈入力欄と感情入力欄が分離され、どちらも必須入力である
2. 感情欄は短文入力を前提にUIガイドが表示される
3. 必須項目が空の場合、送信時にエラーメッセージが表示される

### US-02 コンピテンシー整合の文章生成
**As a** エンジニアユーザー  
**I want** 感情と意図を保ったまま、自然なビジネス文に変換してほしい  
**So that** そのまま送れる返信を短時間で得られる

**Acceptance Criteria**
1. 変換結果はビジネス文として自然な日本語で出力される
2. 変換結果にユーザーの意図が反映されていると判断できる
3. 変換処理中はローディング表示があり、10秒以内に結果または失敗を返す
4. 変換失敗時は再試行可能なエラーメッセージを表示する

### US-03 出力文の即時利用
**As a** エンジニアユーザー  
**I want** 生成文をワンクリックでコピーしたい  
**So that** Slack返信へ即時転用できる

**Acceptance Criteria**
1. 生成結果にコピー操作が提供される
2. コピー成功時に視覚的フィードバックが表示される
3. 生成結果が空のときはコピー操作が無効化される

## Epic 2: Slack連携体験

### US-04 Slack返信補助
**As a** エンジニアユーザー  
**I want** Slack上で返信候補を得たい  
**So that** 会話コンテキストのまま迅速に返信できる

**Acceptance Criteria**
1. Slack連携エンドポイントが入力を受け取り、返信候補を生成できる
2. 返信候補はSlackで扱えるテキスト形式で返される
3. Slack側連携エラー時は、原因が分かるログ/メッセージを返す

### US-05 Web/API共通ロジック
**As a** 運用担当  
**I want** Web経由とSlack経由で同等品質の変換結果を得たい  
**So that** チャネル差異による品質ブレを抑えられる

**Acceptance Criteria**
1. 変換ロジックは共通サービスとして実装される
2. Web/API/Slackの各入口で同一の入力検証ルールを適用する
3. 片系統のみ失敗する不整合ケースが検知可能なログを持つ

## Epic 3: 履歴・個人最適化

### US-06 履歴保存
**As a** エンジニアユーザー  
**I want** 入力内容と変換結果を履歴として保存したい  
**So that** 後で見返して再利用できる

**Acceptance Criteria**
1. 文脈、感情、生成結果、作成時刻がユーザー単位で保存される
2. 保存失敗時はユーザーに通知され、再試行できる
3. 履歴保存は認証ユーザーに限定される

### US-07 履歴参照
**As a** エンジニアユーザー  
**I want** 過去の履歴を一覧で確認したい  
**So that** 過去の表現を再利用し、作成時間を短縮できる

**Acceptance Criteria**
1. 新しい順に履歴一覧を表示できる
2. 各履歴から生成文を再コピーできる
3. 他ユーザーの履歴にはアクセスできない

### US-08 OAuth認証
**As a** エンジニアユーザー  
**I want** OAuthでログインしたい  
**So that** 自分の履歴を安全に扱える

**Acceptance Criteria**
1. OAuthログインフローを開始/完了できる
2. ログイン後にユーザー識別子がセッションに保持される
3. 未認証時に履歴APIへアクセスすると認証エラーになる

## Epic 4: 継続利用を生む体験

### US-09 個人文体への最適化（MVP最小）
**As a** エンジニアユーザー  
**I want** 使うほど自分に近い言い回しで提案してほしい  
**So that** 「これなしでは書きたくない」体験につながる

**Acceptance Criteria**
1. 過去履歴を参照した生成オプションを有効化できる
2. 生成文のトーンが過去履歴と大きく乖離しない
3. 最適化未利用時は通常生成へフォールバックできる

## INVESTセルフチェック
- **Independent**: 各ストーリーは入力/変換/Slack/履歴/認証で独立性を維持
- **Negotiable**: 実装詳細（Slack UI形式、最適化手法）は後続で調整可能
- **Valuable**: すべてユーザー価値またはMVP成立要件に直結
- **Estimable**: 各ストーリーはAPI/UI/データで見積可能な粒度
- **Small**: MVP期間で分割実装可能なサイズ
- **Testable**: 受け入れ条件により正常系/主要異常系を検証可能

## ストーリーとペルソナ対応
- 主ペルソナ（エンジニア）: US-01〜US-09 全体
- 副ペルソナ（評価者）: US-02, US-04, US-09
- 追加ターゲット（非エンジニア当事者）: US-01, US-02, US-03, US-06, US-07
