# Competency Examples（デモ用・合成サンプル）

本ファイルは **特定企業の人事制度や社内資料を転記したものではありません。**  
ハッカソン向け製品（例: `requirements.md` のコンピテンシー入力欄）で「どんな文章を想定しているか」を示すために、**一般に通用的な評価制度の体裁でこちら側が新規に整えた例**です。実在の評価制度・制度名・等級設計とは対応させていません。

多くのサービス公開資料にある **「複数の評価者から能力・態度を集める（いわゆる360度評価）」「コンピテンシーを軸に多面的に評価する」** といった説明レイヤーのイメージに近づける用途向けです。

---

## 1. 評価の考え方（デモ用途の説明文・例）

組織の **目的や価値観から導いた「成果につながる行動特性」** をコンピテンシーとして定義し、日々の協業との関連が高いメンバーからのフィードバックなどを組み合わせて見立てる、というタイプの設計を想定したときの説明例です（文言はすべて本リポジトリ用に作成）。

---

## 2. 複合スキル（4領域）— 概要定義サンプル

いわゆる「大きな枠」の例。製品側ではユーザーがこの塊ごと細かく貼らず、**自由記述で近いニュアンスを入力**する前提でも問題ありません。

| 領域名 | 概要（製品入力の粒度イメージ） |
| :--- | :--- |
| **経営視点** | ビジネス的文脈を読み取りつつ、中長期的な視点で全体として筋の良い方針を描いたり、課題の芽を構造的に見つけたりできること。 |
| **業務遂行** | ゴール設定と優先順位が妥当で、計画〜実行〜振り返りを回して成果に近づけられること。 |
| **コラボレーション** | 利害や情報の非対称があるなかでも対話と調整を進め、心理的安全性が損なわれにくい進め方で合意や相乗を作れること。 |
| **成長支援** | 相手の状況や志向を捉えつつ、学びや改善が続くような支援（対話・機会設計など）ができること。 |

---

## 3. 基礎スキル・マインド（分解例・すべて合成）

「複合スキル」をさらに指標へ落とすときの一例です。**名称や定義文はすべてデモ向けに新規作成**しており、特定の評価制度の正式定義とは一致させていません。

### 経営視点に紐づく例

| 指標の例（仮） | 定義の例（仮） |
| :--- | :--- |
| **構造化思考** | 前提・因果・論点を切り分け、他人にも追いやすい形で説明できる。 |
| **収益・リスク感度** | 数字と制約を踏まえ、判断の妥当性を言語化できる。 |
| **改革案の創出** | 既存手順への依存だけで済まず、仕組み面の改良案を出せる。 |

### 業務遂行に紐づく例

| 指標の例（仮） | 定義の例（仮） |
| :--- | :--- |
| **目標達成への粘り強さ** | 障害があっても打ち手を変えつつやり遂げる。 |
| **役割に必要な技術活用** | 担当領域の手法・ツールを使いこなし、標準より速い復元（リカバリー）ができる。 |

### コラボレーションに紐づく例

| 指標の例（仮） | 定義の例（仮） |
| :--- | :--- |
| **合意形成の対話** | 自分の主張だけに固定せず、より良い結論へ議論を更新できる。 |
| **視点の切り替え** | ステークホルダーごとの制約や意図を踏まえた提案ができる。 |

### 成長支援に紐づく例

| 指標の例（仮） | 定義の例（仮） |
| :--- | :--- |
| **傾聴と気づき支援** | まず整理を手伝い、相手が次のアクションを選びやすい対話ができる。 |
| **機会設計・振り返り** | 練習や委譲などの機会を用意し、学びが再現できる形でフィードバックできる。 |

### 参考：横断テーマとして足し込むことの多い項目（例）

人事制度では別枠になりがちですが、アプリ入力では短文で混ざることがあるので、**デモ用の追加指標例**だけ置いておきます。

| 指標の例（仮） | 定義の例（仮） |
| :--- | :--- |
| **セルフマネジメント** | 時間・気力・対人ストレスなどを自分で整え、約束どおり運用できる。 |
| **価値観の実践** | 組織の合意している価値観と齟齬のない選択が多い。 |
| **知の共有・AI活用** | 検索や生成AIを含む手段で調査や整理を効率化し、再現しやすい形で共有する。 |

## 4. 「強み／伸びしろ」の4段階ラベル例（ヒアリング用）

評価制度ごとに表現は違うため、このファイルでは **機能デモ／プロンプト例** に使いやすいラベルを置いておきます。

| ラベル | 意味 |
| :--- | :--- |
| **強みとして明確** | 自分の強み側に並びやすい |
| **水準として十分** | 役割上、不足を感じにくい |
| **改善余地あり** | 伸ばすと体感の負荷が減る／成果が増えやすい |
| **要フォローの領域** | 周囲の支えなく進めるには難しさがある |

※ 実務では「その職級で何が期待されているか」を前提に他者評価が入る、など制度固有の決まりがありますが、**このファイルには特定制度の運用文言は載せません。**

---

## 5. 職級に応じた「期待の幅」（超要約ダミー）

年収や社内称号に紐づくような **細かなマトリクスは載せません**（識別性が高く、デモにも不要になりがちなため）。

代わりに、製品側でユーザーが頭の中にあるイメージを文章化しやすいよう、**6段のうち最初と最後だけ**抽象的に置いた例です。中間はユーザー入力に任せる想定です。

| レベル（仮） | 期待の広がりの例（完全に抽象的なダミー） |
| :--- | :--- |
| **Lv.1（例）** | 自分担当のゴール・制約・関係者を説明できる。指示された範囲で進捗と品質を揃えられる。 |
| **Lv.6（例）** | 組織横断で方針の整合を取り、持続可能な運用まで含めて前に進める。外部ステークホルダーとも期待値をすり合わせられる。 |

## `requirements.md` での使い方

- コンピテンシー入力欄には、ユーザーが実際に持っている評価基準のPDFや社内サイトの文言をコピペする用途を想定しており、このファイルは **初期デモ／サンプル文の下敷き** として切り貼りして構いません。
- 競合サービスや特定企業の制度を再現する意図はありません。
